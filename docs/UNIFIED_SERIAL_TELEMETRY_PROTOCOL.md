# 果园巡检机器人电控端串口遥测协议

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 文档名称 | 果园巡检机器人电控端串口遥测协议 |
| 协议代号 | OP-Telemetry |
| 协议版本 | V1.0 |
| 文档状态 | 待双方确认 |
| 适用平台 | 甲方 A、甲方 B 通用 |
| 发送方 | 机器人电控端 |
| 接收方 | 视觉主机 |
| 通信方式 | 电控端 -> 视觉主机，单向通信 |
| 编制日期 | 2026-07-15 |

本协议作为电控端串口程序的开发、打包和联调依据。电控端负责采集真实机器人数据，按照本文规定的字段顺序、数据类型、单位和校验算法生成二进制数据包，并通过串口周期发送给视觉主机。

本协议不包含视觉主机向电控端发送的数据，不定义控制指令、病害结果返回、ACK 应答或重传请求。电控端发送后不等待视觉主机回复。

## 2. 通信职责

### 2.1 电控端职责

- 采集机器人 ID、运行状态、路线、路径点、果树、GPS、速度、方向、电池和故障等真实数据。
- 按照本协议规定的类型、单位和字节序打包数据。
- 正确设置字段有效位，不得用随机值代替不可用数据。
- 维护发送包序号并计算 CRC16。
- 按固定周期持续发送，在关键状态变化时立即补发。

### 2.2 视觉主机行为

- 接收串口字节流，处理拆包和粘包。
- 校验帧头、版本、类型、长度、帧尾、CRC 和字段范围。
- 保存最近一包有效遥测数据，并检测链路超时。
- 不向电控端返回数据或应答。

电控端只发送本文定义的机器人遥测数据，不需要了解甲方平台协议，也不得发送 HTTP/UDP 地址、果园前缀或其他平台专用内容。

## 3. 物理与串口参数

| 参数 | 约定 |
| --- | --- |
| 波特率 | 9600 bit/s |
| 数据位 | 8 bit |
| 校验位 | 无 |
| 停止位 | 1 bit |
| 流控 | 无 |
| 字节格式 | 8N1 |
| 推荐发送频率 | 5 Hz |
| 允许最高周期频率 | 10 Hz |
| 字节序 | 多字节整数均为大端序 |

电控端应按固定周期发送完整遥测包。机器人状态、果树编号或故障状态发生变化时，应立即额外发送一包，不必等待下一个周期。

本协议定义串口应用层数据格式，不限定电气接口是 3.3 V TTL、RS-232、RS-485 或 USB 转串口。双方接线前必须单独确认电压标准和收发器类型，禁止将 RS-232 电平直接连接到 TTL UART。

## 4. 数据类型约定

| 类型 | 长度 | 说明 |
| --- | ---: | --- |
| `uint8` | 1 字节 | 无符号 8 位整数 |
| `uint16` | 2 字节 | 无符号 16 位整数，大端序 |
| `uint32` | 4 字节 | 无符号 32 位整数，大端序 |
| `int32` | 4 字节 | 有符号 32 位补码整数，大端序 |

禁止直接发送 C 语言结构体内存。编译器可能插入对齐字节，且 MCU 本机字节序可能与协议不同。发送端必须按照本协议规定的字节偏移逐字段写入缓冲区。

## 5. 完整帧格式

V1.0 每个数据包固定为 58 字节。

| 绝对偏移 | 长度 | 字段 | 类型 | 固定值或说明 |
| ---: | ---: | --- | --- | --- |
| 0 | 2 | 帧头 | bytes | 固定 `AA 55` |
| 2 | 1 | 协议版本 | `uint8` | 固定 `01` |
| 3 | 1 | 消息类型 | `uint8` | 固定 `01`，机器人遥测 |
| 4 | 2 | 包序号 | `uint16` | 每发送一包加 1 |
| 6 | 2 | 数据区长度 | `uint16` | 固定 `00 2E`，即 46 字节 |
| 8 | 46 | 遥测数据区 | bytes | 见第 6 节 |
| 54 | 2 | CRC16 | `uint16` | 高字节在前 |
| 56 | 2 | 帧尾 | bytes | 固定 `0D 0A` |

包序号由电控端维护，初始值可以为 0。达到 `65535` 后，下一个序号回绕到 0。视觉主机使用包序号检测重复包、丢包和电控重启，但周期遥测不要求补发历史包。

### 6.1 机器人状态

| 值 | 状态 | 说明 |
| ---: | --- | --- |
| 0 | 待机/停止 | 设备在线，但当前不行驶 |
| 1 | 正常巡检 | 正常行驶或执行巡检任务 |
| 2 | 到树作业 | 已到果树附近，正在拍摄或作业 |
| 3 | 充电 | 正在充电 |
| 4 | 暂停/人工模式 | 任务暂停或人工接管 |
| 255 | 故障 | 存在需要上报的故障 |

平台状态码与本表不一致时，由视觉主机的平台适配层转换。电控端始终只使用本表定义的通用状态码。

### 6.2 字段有效位

`valid_flags` 的位定义如下：

| 位 | 掩码 | 对应数据 |
| ---: | ---: | --- |
| bit0 | `0x0001` | GPS 定位数据 |
| bit1 | `0x0002` | 路线和路径点 |
| bit2 | `0x0004` | 三个果树编号和拍摄侧 |
| bit3 | `0x0008` | 方位角 |
| bit4 | `0x0010` | 速度 |
| bit5 | `0x0020` | 相机高度 |
| bit6 | `0x0040` | 电池电压 |
| bit7 | `0x0080` | SOC |
| bit8 | `0x0100` | UTC 时间 |
| bit9 | `0x0200` | 故障码 |
| bit10~15 | - | 保留，必须为 0 |

某项数据不可用时，电控端必须清除对应有效位，并将对应字段全部置零。

数值为 0 不等于数据无效。例如速度为 0、电量为 0、海拔为 0 都可能是真实数据。视觉主机应以 `valid_flags` 为准判断有效性。

### 6.3 果树编号和拍摄侧

| `camera_side` | 含义 |
| ---: | --- |
| 0 | 当前拍摄侧未知或当前没有目标果树 |
| 1 | 左侧 |
| 2 | 右侧 |

果树编号按无符号 16 位数值发送，不发送 ASCII 字符串：

```text
显示编号 0001 = 数值 1 = 线上字节 00 01
显示编号 0256 = 数值 256 = 线上字节 01 00
数值 0 = 当前没有对应果树
```

当果树检测功能正常、但当前没有果树时，`valid_flags.bit2` 应为 1，三个果树编号和 `camera_side` 均为 0。这样可以区分“当前没有树”和“果树数据源故障”。

当 `camera_side=1` 时，`current_tree_id` 原则上应等于 `left_tree_id`；当 `camera_side=2` 时，原则上应等于 `right_tree_id`。无法确定当前图像对应哪棵树时，`current_tree_id` 必须为 0，避免产生错误的果树关联。

### 6.4 GPS 定位质量

| 值 | 含义 |
| ---: | --- |
| 0 | 无有效定位 |
| 1 | 单点定位 |
| 2 | 差分定位 |
| 4 | RTK 固定解 |
| 5 | RTK 浮点解 |

只有 `valid_flags.bit0=1` 且 `gps_fix` 不为 0 时，视觉主机才将 GPS 坐标视为有效定位。

## 7. CRC16 校验

CRC 使用 `CRC-16/CCITT-FALSE`：

| 参数 | 值 |
| --- | --- |
| 多项式 `Poly` | `0x1021` |
| 初始值 `Init` | `0xFFFF` |
| 输入反转 `RefIn` | `false` |
| 输出反转 `RefOut` | `false` |
| 结果异或 `XorOut` | `0x0000` |

校验范围为完整数据包的字节 2~53，即从协议版本开始，依次包含消息类型、包序号、数据区长度和全部 46 字节遥测数据。不包含帧头、CRC 字段和帧尾。

CRC 结果以大端序发送，即先发送 CRC 高字节，再发送 CRC 低字节。

参考 C 实现：

```c
#include <stddef.h>
#include <stdint.h>

uint16_t crc16_ccitt_false(const uint8_t *data, size_t length)
{
    uint16_t crc = 0xFFFF;

    for (size_t i = 0; i < length; ++i) {
        crc ^= (uint16_t)data[i] << 8;
        for (uint8_t bit = 0; bit < 8; ++bit) {
            if ((crc & 0x8000U) != 0U) {
                crc = (uint16_t)((crc << 1) ^ 0x1021U);
            } else {
                crc = (uint16_t)(crc << 1);
            }
        }
    }

    return crc;
}
```

## 9. 视觉主机接收规则

视觉主机按以下顺序解析串口字节流：

1. 在接收缓冲区中查找连续帧头 `AA 55`。
2. 至少收到 8 字节后，读取版本、类型和数据区长度。
3. V1.0 只接受版本 `01`、类型 `01`、数据区长度 `46`。
4. 等待接收完整的 58 字节，不得假设一次串口读取等于一包。
5. 检查帧尾 `0D 0A`。
6. 计算并比较 CRC16。
7. 检查字段范围和字段间一致性。
8. 校验成功后，原子替换最新遥测快照。

串口可能出现拆包、粘包、噪声或丢字节。一次读取可能只有半包，也可能包含多包。CRC 或字段校验失败时，视觉主机丢弃错误候选帧，并继续从后续字节中寻找下一个 `AA 55`，不得清空全部接收缓冲区。

视觉主机收到重复包序号时可以忽略重复包。发现包序号跳变时只记录丢包统计，不要求电控补发历史周期包。

## 10. 超时和异常处理

- 最近一包有效遥测的接收时间超过 1 秒时，视觉主机将电控链路标记为失效。
- 链路失效后不得将随机值、估算值或历史值作为实时数据使用。
- 可以保留最后一包数据用于界面显示和故障诊断，但必须同时标记为过期，不得作为有效实时数据上报。
- CRC 错误、长度错误、版本错误和字段越界的数据包均不得更新最新遥测快照。
- 本协议为单向协议，视觉主机不发送 ACK，电控端无须等待应答。

## 11. 电控端实现要求

1. 每次发送必须是完整的 58 字节数据包，不得省略固定字段。
2. 所有多字节整数必须按大端序逐字节写入发送缓冲区。
3. 每发送一包，包序号增加 1；电控重启后允许从 0 重新开始。
4. 正常情况下按 5 Hz 连续发送，串口发送函数不得长期阻塞电控主循环。
5. 状态、果树编号或故障码发生变化时，应立即发送一包最新数据。
6. 电控端不等待 ACK；未收到视觉端数据不影响下一周期继续发送。
7. 数据不可用时必须清除对应有效位并发送 0，禁止填写随机数、估算值或未初始化内存。
8. UTC 时间戳必须使用 UTC Unix 秒，不得直接发送本地时区秒数。
9. `robot_id`、路线编号、路径点编号和果树编号的来源必须稳定，设备重启后不得无规则变化。
10. `fault_code` 的非零值由电控方提供独立对照表；没有故障时发送 0。
11. 发送前必须计算 CRC16，禁止发送 CRC 未更新的旧缓冲区。
12. 电控端自测程序必须能够生成与第 8 节完全一致的标准测试包。

## 12. 联调验收项目

双方至少完成以下测试：

1. 使用第 8 节标准向量核对 58 字节内容和 CRC16。
2. 连续发送 1000 包，核对接收包数、CRC 错误数和序号跳变数。
3. 将一包拆成多次写入，确认视觉端可以正确组包。
4. 一次连续写入多包，确认视觉端可以处理粘包。
5. 修改任意一个数据字节但不更新 CRC，确认视觉端拒绝该包。
6. 发送 `left_tree_id=1`，确认线上字节为 `00 01`，而不是 ASCII `30 30 30 31`。
7. 发送“果树数据有效但当前无树”：`bit2=1` 且果树字段为 0。
8. 发送“果树数据源失效”：`bit2=0` 且果树字段为 0。
9. 发送负经度或负海拔，确认有符号整数解析正确。
10. 停止发送超过 1 秒，确认视觉端进入遥测失效状态且不使用过期数据。
11. 恢复发送，确认视觉端自动恢复并采用最新真实数据。
12. 将包序号从 `65535` 回绕到 `0`，确认接收端可以继续接收。
13. 改变机器人状态、果树编号和故障码，确认电控端立即发送最新数据。
14. 电控端断电重启后从序号 0 恢复发送，确认视觉端可以自动恢复。

## 14. 版本兼容规则

- V1.0 的数据区固定为 46 字节，不得直接在末尾追加字段。
- 增加新字段时，应升级协议版本或定义新的消息类型。
- 接收端遇到不支持的版本或消息类型时，应丢弃该包并记录统计。
- 保留字段在未正式定义前必须发送 0。
